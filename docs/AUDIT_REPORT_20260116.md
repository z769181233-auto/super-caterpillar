## 三、 商业落地最佳方案 (Implementation Plan)

针对 **“导入300万-1500万字小说并产出视频”** 的核心诉求，现有架构无法满足。必须立即暂停业务功能开发，执行 **基础设施重构**。

### 🚀 Stage 4: Industrial Scale (工业化扩容计划)

#### 第一步：重构 CE06 为“流式/扇出”架构 (2周 - 必须优先)

**这是处理 1500万字的唯一路径。**

1.  **Blob Upload**: 小说不再随 API Body 传输，改为上传 S3，Worker 流式读取。
2.  **Lazy Scanning**: `CE06_SCAN` 仅扫描目录 (Table of Contents)，不解析正文，快速生成 `Season/Episode` 骨架 (几秒内完成)。
3.  **Job Fan-out (扇出)**:
    - 以 `Episode` (章节) 为单位拆分任务。
    - 一本 3000 章的小说 -> 自动裂变为 3000 个 `NOVEL_ANALYSIS_CHUNK` 并行任务。
    - 利用 BullMQ + Worker 集群并行消化，独立入库。
4.  **Resumable**: 单个章节失败不影响整书，支持断点重试。

#### 第二步：建立标准化引擎底座 (Engine Standard)

1.  定义 `IEngine` 标准接口 (Input/Output/Cost/Audit)。
2.  将散落在 Worker 中的 CE09/FFmpeg 逻辑归位。
3.  为后续 50+ 引擎的快速接入铺平道路 (Plugin 模式)。

#### 第三步：质量与一致性 (Quality)

1.  **CE01 (Reference Sheet)**: 必须开发。对于长篇小说，主角脸部一致性是体验核心。
2.  **Prompt Finetuning**: 针对主流网文类型 (玄幻/都市) 调优 CE04 的 Prompt 模板。

---

## 四、 风险处理与建议

1.  **立即冻结**：停止开发新的 P2 引擎（如翻译、风格化），资源全部投入 **CE06 重构**。
2.  **严控 Bypass**：生产环境 (`PRODUCTION_MODE=1`) 严禁使用 Bypass Token，否则产出垃圾视频会击穿用户信任（已在代码层面封死）。
3.  **技术债清理**：`apps/workers` 中的大单体函数必须拆解，否则无法多人协作维护 60+ 引擎。

**一句话总结**:
咱们的车架子（Pipeline）搭好了，发动机（Render）也能转了，但油箱入口（Import）太小，加 300万升油（小说）会把车撑爆。
**下一步：把油箱改成空中加油（流式/分片），然后就能起飞了。**
