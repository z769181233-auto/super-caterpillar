# doc_alignment_ce11_and_pipeline.md

本报告基于《Gemini 开发启动指令书》及相关规范文档，对当前系统实现与商业级要求进行对齐分析。

## 1. 对齐基准依据

- **DBSpec V1.1**: 《毛毛虫宇宙\_数据库设计说明书》
- **APISpec V1.1**: 《毛毛虫宇宙\_API设计规范》
- **EngineSpec V1.1**: 《毛毛虫宇宙\_引擎体系说明书》
- **Instruction**: 《Gemini 开发启动指令书*完整版*正式版》

## 2. 差距清单 (Gap Analysis)

### 2.1 数据库与资产归属 (DBSpec)

| 规范条目          | 文档要求                                         | 当前实现                         | 状态 | 差距/风险                                               |
| :---------------- | :----------------------------------------------- | :------------------------------- | :--- | :------------------------------------------------------ |
| **Asset 实体**    | project_id, ownerType, ownerId, shotId(nullable) | shotId 关联已移除，仅靠 ownerId  | ❌   | 缺乏数据库级强一致性约束（FK）；不符合“保留 FK”的要求。 |
| **Asset 索引**    | (ownerType, ownerId, type), shotId               | 缺少 shotId 索引                 | ⚠️   | 查询性能风险。                                          |
| **AuditLog 结构** | payload 包含 resource_type, resource_id, ip, ua  | 字段分散在顶层，payload 内容不全 | ⚠️   | 审计详情不符合 V1.1 规范要求。                          |
| **Security Flag** | video_jobs 需含 security_processed               | shot_jobs 无此显式标志           | ❌   | 无法在 DB 层级追踪安全流水线闭环状态。                  |

### 2.2 API 安全与契约 (APISpec)

| 规范条目         | 文档要求                                       | 当前实现                         | 状态 | 差距/风险                                                              |
| :--------------- | :--------------------------------------------- | :------------------------------- | :--- | :--------------------------------------------------------------------- |
| **HMAC 算法**    | HMAC-SHA256(apiKey + nonce + timestamp + body) | HMAC-SHA256(method + path + ...) | ⚠️   | 算法细节与 Stage1 摘录描述有偏离（虽然代码目前更安全，但需对齐规范）。 |
| **/story/parse** | 核心入口接口，支持小说上传与解析               | 尚未实现或未对齐 Spec 路径       | ❌   | 缺乏主链路入口点。                                                     |
| **4003/4004**    | 签名/重放错误必须使用特定错误码                | 部分使用 400/401 默认返回        | ❌   | 错误处理不符合 API 标准规范。                                          |

### 2.3 引擎体系与 DAG (EngineSpec)

| 规范条目               | 文档要求                            | 当前实现                                       | 状态 | 差距/风险                                             |
| :--------------------- | :---------------------------------- | :--------------------------------------------- | :--- | :---------------------------------------------------- |
| **PIPELINE_E2E_VIDEO** | CE06 -> CE03 -> CE04 -> ... -> CE09 | 仅实现到 CE04，且逻辑硬编码在 JobService       | ⚠️   | 缺乏统一的 DAG 描述逻辑；闭环未延伸至 CE09 安全环节。 |
| **audit_trail 强制**   | 每个引擎步骤必须写 audit_trail      | 部分处理器（如 CE03/04）审计覆盖不足           | ⚠️   | 生产过程不可审计风险。                                |
| **I/O 对齐**           | payload / output 严格对齐 Spec      | 目前部分字段命名较随意（如 previousJobResult） | ❌   | 不利于引擎解耦与扩展。                                |

## 3. 待补强建议 (Planning Input)

1. **Restore FK**: 在 `Asset` 模型中恢复 `shotId` (nullable) 并在迁移中添加针对 `ownerType='SHOT'` 的约束。
2. **Standardize HMAC**: 统一 API 与 Worker 的签名算法，严格按照 `APISpec` V1.1 定义或同步更新规范。
3. **Formalize DAG**: 将 `PIPELINE_E2E_VIDEO` 的流转逻辑从硬编码改为基于任务状态的回调机制，确保覆盖至 `CE09`。
4. **Audit Audit**: 重新校对全链路 `auditLog` 调用，确保满足 V1.1 的字段要求。
